## 1. IntroduĂ§ĂŁo

O **Factory Method** Ă© um padrĂŁo de projeto **criacional** descrito pela Gang of Four (GoF). Seu objetivo Ă© definir uma interface para criar um objeto, mas **deixar que as subclasses decidam qual classe concreta instanciar**. Esse padrĂŁo Ă© especialmente Ăştil quando:

* O cĂłdigo cliente **nĂŁo deve conhecer** as classes concretas que serĂŁo instanciadas.
* O sistema precisa estar **aberto Ă  extensĂŁo** (novos tipos de produto) sem alterar o cĂłdigo existente â€” **OCP**.


## 2. MotivaĂ§ĂŁo no OrganizeSeuGrupo

## 3. Estrutura do PadrĂŁo (mapeamento ao projeto)

| Papel GoF | Classe no projeto | Responsabilidade |
| :-------- | :---------------- | :--------------- |
| **Product** | `Material` (interface) | Define o contrato pĂşblico (`visualizar(): void`). |
| **AbstractProduct** | `MaterialBase` (classe abstrata) | Centraliza o estado comum (`id`, `nome`) e mantĂ©m `visualizar()` abstrato. |
| **ConcreteProduct** | `MaterialPDF`, `MaterialVideo` | Materiais concretos com seus atributos e implementaĂ§ĂŁo de `visualizar()`. |
| **Creator** | `MaterialFactory` (classe abstrata) | Declara o *factory method* `criarMaterial(...)` e fornece a operaĂ§ĂŁo concreta `processarNovoMaterial(...)`. |
| **ConcreteCreator** | `PDFFactory`, `VideoFactory` | Implementam o *factory method*, instanciando o `ConcreteProduct` correspondente. |

### Fluxo de execuĂ§ĂŁo

1. O cliente instancia a fĂˇbrica concreta (`new PDFFactory()` ou `new VideoFactory()`).
2. Chama `processarNovoMaterial(id, nome, extraData)` na fĂˇbrica.
3. Esse mĂ©todo invoca internamente o *factory method* `criarMaterial(...)`, que devolve um `Material` (referĂŞncia polimĂłrfica).
4. A fĂˇbrica simula a persistĂŞncia (`[DB] Salvando...`) e chama `material.visualizar()`.
5. O polimorfismo resolve, em tempo de execuĂ§ĂŁo, qual `visualizar()` serĂˇ executado â€” sem que o cliente conheĂ§a a classe concreta.

### 5.1. Product â€” interface `Material`

```typescript
// Product 
interface Material {
    visualizar(): void;

```

### 5.2. AbstractProduct â€” classe abstrata `MaterialBase`

```typescript
// AbstractProduct 
abstract class MaterialBase implements Material {
    private id: number;
    private nome: string;

    constructor(id: number, nome: string) {
        this.id = id;
        this.nome = nome;
    }

    protected getId(): number {
        return this.id;
    }

    protected getNome(): string {
        return this.nome;
    }

    public abstract visualizar(): void;
}
```

> **Por que separar `Material` (interface) de `MaterialBase` (classe abstrata)?**
> A interface define o **contrato externo** â€” Ă© o ponto de acoplamento usado pelo restante do sistema (DIP). A classe abstrata isola o **estado comum** entre os produtos, evitando duplicaĂ§ĂŁo (SRP). Manter `visualizar()` abstrato em `MaterialBase` obriga cada produto concreto a implementar seu prĂłprio comportamento de visualizaĂ§ĂŁo.

### 5.3. ConcreteProduct â€” `MaterialPDF`

### 5.5. Creator â€” classe abstrata `MaterialFactory`

```typescript
// Creator

    public abstract criarMaterial(
        id: number,
        nome: string,
        extraData: any

    public processarNovoMaterial(
        id: number,
        nome: string,

```

## 8. Senso CrĂ­tico

### 8.1. BenefĂ­cios obtidos no OrganizeSeuGrupo

* **Extensibilidade (OCP):** adicionar um `MaterialPodcast` exige apenas criar duas classes novas (`MaterialPodcast` e `PodcastFactory`). Nenhum cĂłdigo existente precisa ser alterado.
* **Baixo acoplamento (DIP):** o cliente depende apenas de `MaterialFactory` e `Material` â€” nĂŁo conhece classes concretas.
* **CoesĂŁo (SRP):** a lĂłgica de criaĂ§ĂŁo fica nas fĂˇbricas; a lĂłgica de visualizaĂ§ĂŁo fica nos produtos. Cada classe tem uma Ăşnica razĂŁo para mudar.
* **Reuso de orquestraĂ§ĂŁo:** `processarNovoMaterial` Ă© definido **uma Ăşnica vez** no `Creator` e reaproveitado por todas as fĂˇbricas concretas.

### 8.2. Custos e ressalvas

* **Mais classes:** o padrĂŁo multiplica o nĂşmero de classes. Em sistemas pequenos ou com poucos tipos de produto, pode ser exagero.
* **IndireĂ§ĂŁo:** desenvolvedores que leem o cĂłdigo pela primeira vez precisam entender o fluxo polimĂłrfico â€” hĂˇ um overhead cognitivo.
* **`extraData: any`:** optamos por um parĂ˘metro genĂ©rico no *factory method* para permitir variaĂ§Ăµes entre tipos. Em produĂ§ĂŁo, valeria refinar com *type guards* ou *generics* para reforĂ§ar a tipagem (`MaterialFactory<TExtra>`).
